# 皮肤护理小程序时序图详细设计

## 4.5.1 AI 皮肤分析与建议生成时序图（图4-10）

### 时序图流程

1. **用户操作**：用户进入肤质分析页面，点击"拍摄并分析"按钮
2. **前端处理**：小程序前端调用相机API拍摄照片
3. **请求发送**：小程序前端向后端发送皮肤测试请求，包含拍摄的照片
4. **后端处理**：
   - 后端控制器接收请求
   - 封装Prompt（包含皮肤照片信息和分析要求）
   - 调用第三方AI服务接口
5. **AI分析**：第三方AI服务（如DeepSeek/阿里云）进行皮肤分析
6. **结果返回**：AI服务返回分析结果（JSON格式）
7. **数据处理**：后端控制器解析JSON数据，提取肤质类型、评分和建议
8. **数据存储**：后端将分析结果存储到数据库
9. **响应返回**：后端返回分析结果给前端
10. **结果展示**：前端展示分析报告

### 时序图层次

| 层次 | 角色 | 操作 |
|------|------|------|
| 用户层 | 用户 | 点击拍摄按钮 |
| 前端层 | 小程序前端 | 调用相机API，上传照片，发送请求，展示结果 |
| 后端层 | 后端控制器 | 接收请求，封装Prompt，调用AI接口，解析结果，存储数据 |
| AI服务层 | 第三方AI服务 | 处理皮肤分析请求，返回分析结果 |
| 数据层 | 数据库 | 存储皮肤分析记录 |

### 代码依据

- 前端代码：`ma/src/pages/skin-test/index.vue` 中的 `takePhoto` 方法
- 后端控制器：`server/api/ma/src/main/java/com/tacomall/apima/controller/ApiSkinController.java`
- 后端服务：`server/api/ma/src/main/java/com/tacomall/apima/service/impl/SkinTestServiceImpl.java`

## 4.5.2 用户登录与鉴权时序图（图4-11）

### 时序图流程

1. **用户操作**：用户打开小程序
2. **前端处理**：小程序前端调用微信登录API获取code
3. **请求发送**：前端向后端发送登录请求，包含code
4. **后端处理**：
   - 后端控制器接收请求
   - 调用微信接口，使用code换取openid
   - 根据openid查询数据库中的用户信息
5. **用户处理**：
   - 如果用户存在，更新用户信息（昵称、头像、登录时间）
   - 如果用户不存在，创建新用户（生成account编号）
6. **响应返回**：后端返回用户信息给前端
7. **前端处理**：前端存储用户信息并进入首页

### 时序图层次

| 层次 | 角色 | 操作 |
|------|------|------|
| 用户层 | 用户 | 打开小程序 |
| 前端层 | 小程序前端 | 调用微信登录API，发送登录请求，存储用户信息 |
| 后端层 | 后端控制器 | 接收请求，调用微信接口，查询/创建用户 |
| 微信接口层 | 微信接口 | 提供code换取openid服务 |
| 数据层 | 数据库 | 存储用户信息 |

### 代码依据

- 前端代码：`ma/src/utils/api.js` 中的 `login` 方法
- 后端控制器：`server/api/ma/src/main/java/com/tacomall/apima/controller/ApiAuthController.java`
- 后端服务：`server/api/ma/src/main/java/com/tacomall/apima/service/impl/AuthServiceImpl.java`

## 4.5.3 每日护肤打卡时序图（图4-12）

### 时序图流程

1. **用户操作**：用户进入打卡页面，加载化妆品列表
2. **用户操作**：用户选择本次使用的护肤品
3. **用户操作**：用户点击"完成打卡"按钮
4. **前端处理**：前端向后端发送打卡请求，包含选择的化妆品ID列表
5. **后端处理**：
   - 后端控制器接收请求，开始事务处理
   - 查询今日是否已打卡
   - 如果今日已打卡，抛出异常并回滚事务
   - 如果今日未打卡，保存打卡主记录
   - 保存关联的化妆品清单（按选择顺序）
   - 更新连续打卡天数
   - 提交事务
6. **响应返回**：后端返回打卡结果给前端
7. **前端处理**：前端显示打卡成功并跳转到打卡详情页面

### 时序图层次

| 层次 | 角色 | 操作 |
|------|------|------|
| 用户层 | 用户 | 选择护肤品，点击打卡按钮 |
| 前端层 | 小程序前端 | 发送打卡请求，显示打卡结果 |
| 后端层 | 后端控制器 | 接收请求，进行事务处理，查询打卡状态，保存记录 |
| 数据层 | 数据库 | 存储打卡记录和关联的化妆品清单 |

### 代码依据

- 前端代码：`ma/src/pages/checkin/index.vue` 中的 `submitCheckin` 方法
- 后端控制器：`server/api/ma/src/main/java/com/tacomall/apima/controller/ApiCheckinController.java`
- 后端服务：`server/api/ma/src/main/java/com/tacomall/apima/service/impl/CheckinServiceImpl.java`

## 4.5.4 系统配置管理时序图（图4-13）

### 时序图流程

1. **管理员操作**：管理员进入系统配置管理页面
2. **前端处理**：前端向后端请求当前系统配置
3. **后端处理**：后端查询数据库获取当前配置
4. **响应返回**：后端返回配置数据给前端
5. **前端处理**：前端显示当前配置并允许修改
6. **管理员操作**：管理员修改配置（AI API Key、Base URL、模型名称、Temperature、Max Token、各种Prompt等）
7. **管理员操作**：管理员点击"保存并更新"按钮
8. **前端处理**：前端向后端发送配置更新请求
9. **后端处理**：
   - 后端控制器接收请求
   - 校验配置格式（如API Key是否合法）
   - 更新数据库中的配置
   - 清除缓存（Redis）或热更新，让AI模型立即生效
10. **响应返回**：后端返回更新结果给前端
11. **前端处理**：前端显示保存成功提示

### 时序图层次

| 层次 | 角色 | 操作 |
|------|------|------|
| 用户层 | 管理员 | 进入配置页面，修改配置，点击保存按钮 |
| 前端层 | 管理员前端 | 请求配置，显示配置，发送更新请求，显示保存结果 |
| 后端层 | 后端控制器 | 接收请求，校验配置，更新数据库，清除缓存 |
| 缓存层 | Redis | 存储配置缓存，接收缓存清除指令 |
| 数据层 | 数据库 | 存储系统配置信息 |

### 代码依据

- 前端代码：`admin/src/pages/skin-assistant/system-config.vue` 中的 `loadConfig` 和 `saveConfig` 方法
- 前端API：`admin/src/utils/skin-assistant-admin-api.js` 中的 `getConfigList` 和 `updateConfig` 方法
- 后端控制器：`server/api/ma/src/main/java/com/tacomall/apima/controller/SystemController.java`
- 后端服务：`server/api/ma/src/main/java/com/tacomall/apima/service/impl/SysConfigServiceImpl.java`

## 时序图绘制方法

### 工具选择

推荐使用以下工具绘制时序图：
1. **Draw.io**：免费、在线、支持多种格式
2. **PlantUML**：基于文本的UML绘制工具
3. **Microsoft Visio**：专业的图表绘制工具

### 绘制步骤

1. **确定参与者**：根据上述层次确定每个时序图的参与者
2. **绘制生命线**：为每个参与者绘制一条垂直的生命线
3. **添加消息**：按照流程顺序，在参与者之间添加消息
4. **标注时间顺序**：使用数字标注消息的时间顺序
5. **添加注释**：对关键步骤添加注释，说明其功能
6. **美化图表**：调整布局，确保图表清晰易读

### 注意事项

1. **箭头方向**：消息箭头应从发送方指向接收方
2. **消息顺序**：严格按照时间顺序排列消息
3. **层次清晰**：确保参与者层次清晰，便于理解
4. **代码对应**：确保时序图与代码实现对应，准确反映实际流程
5. **重点突出**：突出每个模块的核心流程和亮点

